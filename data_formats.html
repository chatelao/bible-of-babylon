Data Format Patterns

BasicTypes

description:

Representation of fundamental data types: Strings, Numbers, and Booleans.

Parameters:

  • string_val: String

  • number_val: String

  • boolean_val: String

  • notes: String

BasicTypes Comparison

Language

String val

Number val

Boolean val

Notes

JsonBasic

 
"Hello"

42 or 3.14

true / false

Standard JSON types; strings must be double-quoted.

XmlBasic

 
<tag>Hello</tag>

<tag>42</tag>

<tag>true</tag>

XML is text-based; types are typically inferred or defined by a schema (XSD).

YamlBasic

 
Hello or 'Hello' or "Hello"

42 or 3.14

true / false (or yes/no, on/off in YAML 1.1)

YAML supports plain, single-quoted, and double-quoted strings.

TomlBasic

 
"Hello"

42 or 3.14

true / false

TOML is strictly typed; strings must be double-quoted.

CsvBasic

 
Hello or "Hello"

42 or 3.14

true / false

CSV values are typically plain text; quoting is used for values containing delimiters or newlines.

FixlengthBasic

 
Fixed width string (e.g., 'Hello     ')

Fixed width number (e.g., '00042')

Fixed width indicator (e.g., 'Y' / 'N')

Data is defined by fixed positions and lengths rather than delimiters.

Collection

description:

Ordered sequence of elements (Arrays, Lists, or Sequences).

Parameters:

  • elements: String

  • syntax: String

  • notes: String

Collection Comparison

Language

Syntax

Notes

JsonCollection

 
[1, "two", true]

Elements are comma-separated and enclosed in square brackets.

XmlCollection

 
<list>
  <item>1</item>
  <item>2</item>
</list>

Collections are typically represented by repeating elements under a parent.

YamlCollection

 
- 1
- two
- true

Uses a dash followed by a space for each element.

TomlCollection

 
[1, 2, 3]

Arrays can contain values of different types since TOML 1.0.0.

CsvCollection

 
1,two,true

A single row represents a collection of fields.

FixlengthCollection

 
Field1Field2Field3

Collections are typically represented by a sequence of fields with pre-defined lengths.

Mapping

description:

Unordered collection of key-value pairs (Objects, Maps, or Dictionaries).

Parameters:

  • entries: String

  • syntax: String

  • notes: String

Mapping Comparison

Language

Syntax

Notes

JsonMapping

 
{"key": "value", "num": 42}

Keys must be double-quoted strings.

XmlMapping

 
<object>
  <key>value</key>
  <num>42</num>
</object>

Mappings are represented as nested elements.

YamlMapping

 
key: value
num: 42

Uses a colon followed by a space to separate key and value.

TomlMapping

 
key = "value"
num = 42

Top-level or grouped using [headers].

CsvMapping

 
Name,Age
Alice,30

Mappings are established by associating header names with column values.

FixlengthMapping

 
Pos 1-10: Name, 11-15: Age

Mappings are established by a schema defining which positions correspond to which fields.

Metadata

description:

Way to attach metadata or attributes to data elements.

Parameters:

  • syntax: String

  • notes: String

Metadata Comparison

Language

Syntax

Notes

JsonMetadata

N/A

JSON does not support metadata or attributes on elements; often simulated using underscore-prefixed keys (e.g., “_metadata”: { … }).

XmlMetadata

 
<element attr="value">Content</element>

XML has native support for attributes on elements.

YamlMetadata

 
!!str "value" or !custom { key: val }

YAML supports tags to specify types or attach metadata to nodes.

TomlMetadata

N/A

TOML does not support per-element metadata, though it uses headers for grouping.

CsvMetadata

N/A

Standard CSV (RFC 4180) does not support metadata or attributes.

FixlengthMetadata

N/A

Fixed-length formats do not natively support metadata or attributes within the data records.

SingleLineComment

description:

Way to add non-executable explanatory text to the data on a single line.

Parameters:

  • syntax: String

  • notes: String

SingleLineComment Comparison

Language

Syntax

Notes

JsonSingleLineComment

N/A

JSON does not natively support comments.

XmlSingleLineComment

 
<!-- comment -->

XML uses <!– –> for comments.

YamlSingleLineComment

 
# comment

YAML uses # for single-line comments.

TomlSingleLineComment

 
# comment

TOML uses # for single-line comments.

CsvSingleLineComment

N/A

Standard CSV does not natively support comments; some variations use #.

FixlengthSingleLineComment

N/A

Fixed-length formats generally do not support comments.

MultiLineComment

description:

Way to add non-executable explanatory text to the data spanning multiple lines.

Parameters:

  • syntax: String

  • notes: String

MultiLineComment Comparison

Language

Syntax

Notes

JsonMultiLineComment

N/A

JSON does not natively support comments.

XmlMultiLineComment

 
<!--
  line 1
  line 2
-->

XML uses the same syntax for single and multi-line comments.

YamlMultiLineComment

 
# line 1
# line 2

YAML does not have a native block comment; it uses multiple single-line comments.

TomlMultiLineComment

 
# line 1
# line 2

TOML only supports single-line comments starting with #.

CsvMultiLineComment

N/A

Standard CSV does not support multi-line comments.

FixlengthMultiLineComment

N/A

Fixed-length formats generally do not support comments.

SchemaLink

description:

Linking a data file to a schema for validation (e.g., JSON Schema, XSD).

Parameters:

  • syntax: String

  • notes: String

SchemaLink Comparison

Language

Syntax

Notes

JsonSchemaLink

 
"\$schema": "http://json-schema.org/draft-07/schema#"

JSON uses the $schema keyword to point to a JSON Schema file.

XmlSchemaLink

 
<root xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
      xsi:schemaLocation="http://example.com schema.xsd">

XML uses the xsi:schemaLocation attribute to link to an XSD.

YamlSchemaLink

 
# yaml-language-server: \$schema=<url_or_path>

YAML often relies on editor-specific comments (like VS Code’s language server) for schema linking.

TomlSchemaLink

N/A

TOML does not have a native or widely standardized way to link to a schema within the file.

CsvSchemaLink

N/A

CSV does not have a native schema linking mechanism; often defined in a sidecar file (e.g., CSVW metadata).

FixlengthSchemaLink

N/A

Schema definition is usually external (e.g., a Copybook or a COBOL file descriptor).

Header

description:

Standardized header or declaration at the beginning of a file or document.

Parameters:

  • syntax: String

  • notes: String

Header Comparison

Language

Syntax

Notes

JsonHeader

N/A

JSON files do not have a standardized header; they start directly with the data (object or array).

XmlHeader

 
<?xml version="1.0" encoding="UTF-8"?>

XML declaration specifies the version and encoding of the document.

YamlHeader

 
%YAML 1.2
---

The %YAML directive (optional) specifies the version, and the triple dash (—) marks the start of a document.

TomlHeader

N/A

TOML does not have a native file-level header or declaration.

CsvHeader

 
Name,Age,City

The first row of a CSV file often contains the column names (headers).

FixlengthHeader

 
H2026052400001

Fixed-length formats often use a ‘Header’ record (e.g., starting with ‘H’) containing metadata like date or batch ID.
